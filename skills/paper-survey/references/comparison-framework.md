# Comparison Framework — 论文对比框架

对比矩阵是 paper-survey 的核心组织工具。所有输出模式都基于同一个对比矩阵。

## 对比维度

### 维度 1：问题定义 (problem_definition)

| 子问题 | 说明 |
|--------|------|
| 输入输出 | 每篇论文的输入是什么？输出是什么？ |
| 假设条件 | 每篇论文依赖什么假设？ |
| 适用场景 | 每篇论文在什么场景下设计的？ |

### 维度 2：方法路径 (method_approach)

| 子问题 | 说明 |
|--------|------|
| 核心思路 | 一句话说清楚解决方法 |
| 技术路线 | 具体做了哪几步？ |
| 关键设计决策 | 为什么这样做？有没有替代设计？ |

### 维度 3：关键设计细节 (key_design)

| 子问题 | 说明 |
|--------|------|
| 核心模块 | 方法中最创新的那个模块 |
| 数学/算法差异 | 公式/算法的核心区别 |
| 工程实现差异 | 实现层面的关键差异（如并行化、内存优化） |

### 维度 4：实验表现 (experimental_results)

| 子问题 | 说明 |
|--------|------|
| Benchmark 覆盖 | 各自在哪些数据集上评测？ |
| 核心指标 | 各自最好的指标是多少？ |
| 效率对比 | 参数量/推理速度/训练成本 |

### 维度 5：适用边界与 Trade-off (trade_off)

| 子问题 | 说明 |
|--------|------|
| 优势 | 每篇论文做什么最好？ |
| 局限 | 每篇论文哪里不行？ |
| 推荐场景 | 什么场景选哪个方法？ |

## 自适应剪枝

不是所有维度都需要展开。根据论文数量和相关性自动选择：

| 论文关系 | 重点维度 |
|----------|---------|
| 同一系列（如 Transformer 改进） | 方法路径 + 关键设计 + 实验对比 |
| 不同方法（如 CNN vs Transformer） | 问题定义 + 方法路径 + Trade-off |
| 综述（一个主题多方向） | 问题定义 + 适用边界 |

## 标注规则

- **相同点**：用 "⟷" 标记，文字用"都""同样"描述
- **差异点**：用 "↗" 标记，文字用"不同在于""而"描述
- **演进关系**：用 "→" 标记，文字用"改进自""扩展了"描述

## 语言规范

对比文字使用统一句式：

| 场景 | 句式 |
|------|------|
| 描述相同 | "所有这些方法都[共同特征]" |
| 描述差异 | "X 的做法是[做法A]，而 Y 的做法是[做法B]" |
| 描述演进 | "从 X 到 Y，核心改进在于[改进点]" |
| 描述优劣 | "X 的优势在于[优势]，代价是[代价]" |

避免：
- ❌ "X 更好 / Y 更差"（主观判断）
- ✅ "X 在[指标]上比 Y 高[数字]，因为[原因]"（事实判断）
